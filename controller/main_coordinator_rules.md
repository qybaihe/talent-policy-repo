# main_coordinator_rules.md

## talent-policy-repo 主线程协调规则

主线程必须承担协调职责，不只是读摘要。

### 核心目标
- 保证 `talent-policy-repo` **持续、稳步推进**
- 不允许 backlog 长时间停滞
- 不允许只产出状态、不产出内容
- 不允许子任务完成后长时间不收口

### 硬性推进要求
- **主线程每 20 分钟必须推进一次**
- 不能因为脚本只产出 blueprint 就停住
- 不能因为没有自动 spawn 就等待
- 只要 backlog 还有未完成城市，主线程就必须持续补位
- 每次介入都必须至少完成以下动作之一：
  1. 收口已完成稿件
  2. 重跑异常稿件
  3. 补满 3 并发
  4. 更新 backlog / state / 协调规则

### backlog 状态流转
统一按以下状态理解：
- `todo`：尚未开始
- `drafting`：已有骨架或半成品，但质量不够
- `deepsearching`：已被主线程拉起，正在深挖
- `done`：已完成并收口到仓库

规则：
- 发现文件存在但质量不达标，优先视为 `rewrite`
- 发现文件质量达标，立即标记 `done`
- `deepsearching` 长时间不变化，不继续等待，直接重跑或换人补位

### 触发介入条件（满足任一条立即介入）
- 20 分钟无新增 commit
- 20 分钟 backlog 的 `done` 不增长
- `controller/deepsearch_state.json` 中 `running` 数量 < 3
- 已识别 rewrite 城市但长时间未处理
- 已完成深挖稿存在但未 commit/push
- backlog 中仍有 `todo / drafting / deepsearching` 状态
- `rewrite_preview` 持续存在但未转化为真实推进

### 介入动作顺序
1. 先收已完成深挖稿并 commit/push
2. 再优先处理 rewrite 城市（骨架/半成品/缺金额或链接）
3. 再补满 3 并发 deepsearch
4. 子任务完成后立刻收口、更新 backlog、清理 running
5. 若脚本未真实拉起任务，主线程直接按 `rewrite_preview` / backlog 手动拉起
6. 若某城市多次失败，记录失败原因并暂时后移，不阻塞整批吞吐

### 城市选择优先级
默认按以下顺序补位：
1. 已在 backlog 中、优先级更高的省会 / 区域中心城市
2. 已有半成品但明显不合格的 `rewrite` 城市
3. 具有明确金额 / 链接 / 区级政策线索、最容易收口的城市
4. 再处理信息更稀疏、需要更保守写法的城市

### 质量底线
每份稿件至少满足：
- 不是 placeholder / 骨架 / 过程说明
- 有明确的城市结论
- 有政策分层或政策结构说明
- 尽量覆盖：补贴 / 安家 / 租房 / 购房 / 创业 / 落户 / 区县差异 / 申请提醒
- 有参考链接
- 没把不确定金额硬写死
- 对不稳定细则统一写 `待复核`

### 异常处理规则
以下情况一律视为异常，不计入有效推进：
- 只返回搜索过程，没有成稿
- 只补少量碎片，没有完整结构
- 仍保留 placeholder 文案
- 没有链接
- 没有金额线索且没有说明“待复核”
- 区县差异完全缺失，且正文也没有解释城市级政策为主

异常处理动作：
1. 不收口
2. 立即重跑或换城市补位
3. 在 state / backlog 中保留问题痕迹，避免下一轮误判为 done

### 主线程收到摘要后必须做的判断
1. 本轮是否有新 commit？
2. backlog done 是否增长？
3. running 是否变化？
4. `rewrite_preview` 是否仍存在？
5. 是否有已生成未提交稿件？
6. 若无推进：立即读取 `controller/deepsearch_state.json` 与 `city-backlog.json`，直接补位/重跑/收口

### 稳步推进的定义
只要 backlog 未清空，主线程就要保持如下节奏：
- 每 20 分钟至少推进一轮
- 尽量维持 3 并发
- 子任务完成就马上收口
- 不追求“看起来自动化”，优先保证真实产出
- 宁可主线程补位，也不要让项目空转
